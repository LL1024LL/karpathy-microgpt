# 公有云 LLM 推理性能调研报告

> 调研时间：2026-03-03
> 数据来源：Artificial Analysis、LLMPerf Leaderboard、各厂商公开数据
> 说明：实际性能受 prompt 长度、并发量、地区、时间段影响波动较大，以下数据为典型值

---

## 一、核心指标说明

| 指标 | 含义 | 对应原理 |
|------|------|----------|
| **TTFT** (Time To First Token) | 发送请求到收到第一个 token 的延迟 | Prefill 阶段耗时（compute-bound） |
| **TPOT** (Time Per Output Token) | 生成每个输出 token 的平均间隔 | Decode 阶段耗时（memory-bound） |
| **输出速度** (tokens/s) | 每秒生成的 token 数，= 1000 / TPOT | 直接体感指标 |

## 四、与推理原理的关联

### 4.1 为什么 Groq 最快

你学过 Decode 阶段的瓶颈是"搬数据"——每生成一个 token 都要把整个模型的权重从 HBM 搬到计算单元。Groq 的方案是：**干脆不用 HBM���把权重全部放在片上 SRAM 里**。

```
传统GPU:  权重在HBM → 搬到SRAM → 计算 → 搬回去    （带宽瓶颈）
Groq LPU: 权重在SRAM → 直接计算                    （无搬运瓶颈）
```

代价是 SRAM 容量小、成本高，所以 Groq 目前只能跑中小模型（8B、70B），405B 暂时跑不了。

### 4.2 为什么 MoE 便宜

MoE 模型（如 DeepSeek V3）：
- 总参数 671B → 知识容量大，质量高
- 激活参数 37B → 每个 token 只激活 37B 的专家，搬运量 ≈ 37B 模型
- Decode 阶段搬运量小 → 同样的 GPU 可以服务更多用户 → 单价低

### 4.3 TTFT 和 Prefill 的关系

```
TTFT ≈ Prefill耗时 + 网络延迟

Prefill（compute-bound）：prompt越长 → 计算量越大 → TTFT越大
小模型：计算量小 → TTFT 20-200ms
大模型：计算量大 → TTFT 500-2000ms
```

### 4.4 输出速度和 Decode 的关系

```
输出速度 = 1000 / TPOT (tokens/s)

Decode（memory-bound）：模型越大 → 搬运权重越多 → TPOT越大 → 速度越慢
8B模型：  搬运少 → 100-1200 tokens/s
70B模型：  搬运多 → 50-400 tokens/s
405B模型：搬运巨多 → 15-120 tokens/s
```

### 4.5 国内厂商的价格战

字节跳动 Doubao-lite 几乎免费（$0.01/1M tokens），靠的是：
- 极致的 Continuous Batching（合并多用户请求，权重只搬一次）
- INT8/INT4 量化（权重体积缩小2-4倍，搬运量减半）
- 大规模 GPU 集群摊薄成本
- 战略性亏损抢市场

## 五、不同场景的推荐选择

| 场景 | 推荐 | 理由 |
|------|------|------|
| **极致延迟（实时对话）** | Groq + Llama 8B | TTFT < 50ms，输出 1000+ tokens/s |
| **性价比优先** | DeepSeek V3 | GPT-4 级质量，1/10 价格 |
| **最高质量** | Claude Sonnet / GPT-4o | 质量最好，价格高 |
| **超长上下文** | Gemini 1.5/2.0 Pro | 100万 token 上下文窗口 |
| **中文场景低成本** | Doubao / Qwen-Turbo | 接近免费 |
| **自部署最快** | vLLM + A100/H100 | 可控，无网络延迟 |
